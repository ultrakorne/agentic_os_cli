# Theme System — Technical

## Architecture

Theme lives in three layers. (1) Main process: `loadTheme()` reads `colors.toml` + `theme.name`, returns a `Theme`; `ThemeWatcher` debounces fs events and re-loads on change. (2) IPC: `theme:get` / `theme:reload` are request/response, `theme:changed` is push from main to all windows. (3) Renderer: `applyTheme` writes each slot to a `--theme-*` CSS custom property on `<html>`; `styles.css` derives semantic `--color-*` tokens via `color-mix(in oklch, …)`. Components reference only the semantic tokens.

## Source Files

| File | Role |
|------|------|
| `src/shared/theme.ts` | `Theme` + `ThemeColors` types shared across main/preload/renderer |
| `src/main/theme/loader.ts` | TOML parsing, fallback palette, `loadTheme()`, `ThemeWatcher` |
| `src/main/ipc.ts` | Registers `theme:get` / `theme:reload`, exposes `broadcastTheme` |
| `src/main/index.ts` | Constructs `ThemeWatcher` on app ready, stops it on `window-all-closed` |
| `src/preload/index.ts` | `window.api.theme.{get, reload, onChange}` |
| `src/renderer/src/theme.ts` | `applyTheme`: maps `Theme.colors` keys to `--theme-*` CSS vars on `<html>` |
| `src/renderer/src/store.ts` | Subscribes to `theme.onChange`, calls `applyTheme`, stores `Theme` in Zustand |
| `src/renderer/src/styles.css` | `@theme` block: derives semantic `--color-*` tokens from `--theme-*` slots |

## Data Model

The on-disk colors file is parsed with a single regex (`^key = "value"$`) — not a full TOML parser. This is deliberate: the file is generated by Omarchy and only ever uses that one form. If Omarchy ever ships TOML tables or arrays, this will silently ignore them, which is fine — only the flat `colorN` / `background` / `foreground` / etc. keys are needed.

The 22-slot `ThemeColors` shape and its `--theme-*` mirrors are listed in `src/renderer/src/theme.ts` (`VAR_MAP`). Source of truth is the type plus that map; do not duplicate the list elsewhere.

## Noteworthy Behavior

- **Per-key fallback, not whole-theme fallback.** `asColors` calls `pick()` per key; a missing single key falls back to the tokyo-night value while the rest of the theme loads from disk. So a partial Omarchy theme still shows mostly the user's palette.
- **The watcher silently no-ops on missing files.** `fs.watch` throws if the path doesn't exist; we catch and skip. On a non-Omarchy system we end up with a fallback theme and zero active watchers — the app still works, the manual reload button still works, it just won't auto-update (because there's nothing to watch).
- **Debounce on the watcher is 80 ms.** When Omarchy switches a theme, `colors.toml` and `theme.name` may both fire; debouncing collapses them into one reload.
- **Two CSS-var layers, one direction.** Components must read `--color-*` (semantic), not `--theme-*` (raw). The mapping in `styles.css` uses `color-mix(in oklch, ...)` for derived tokens like `--color-fg-dim`, which keeps them perceptually consistent across radically different palettes.
- **`html.dataset.themeName` and `themeSource`** are set by `applyTheme`. Useful for theme-specific tweaks via attribute selectors if a particular palette needs a nudge, and for debugging.
- **No reduced-motion concerns from the theme itself**, but `styles.css` does honor `prefers-reduced-motion` for the `pulse-soft` animation used on running rows. Worth knowing because that animation uses `--color-cool`, which comes from the theme.

## Dependencies

- `node:fs` (promises + `watch`) — file IO and change watching
- `node:os.homedir()` — locating `~/.config/omarchy/`
- Tailwind 4 `@theme` — derives semantic colors from raw slots
- Omarchy (runtime, optional) — provides the theme file; absence triggers fallback
